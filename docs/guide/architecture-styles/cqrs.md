# <a name="cqrs-architecture-style"></a><span data-ttu-id="ddb85-103">Style d’architecture CQRS</span><span class="sxs-lookup"><span data-stu-id="ddb85-103">CQRS architecture style</span></span>

<span data-ttu-id="ddb85-104">Le modèle CQRS (Command and Query Responsibility Segregation, répartition des responsabilités entre les commandes et les requêtes) est un style d’architecture qui sépare les opérations de lecture des opérations d’écriture.</span><span class="sxs-lookup"><span data-stu-id="ddb85-104">Command and Query Responsibility Segregation (CQRS) is an architecture style that separates read operations from write operations.</span></span> 

![](./images/cqrs-logical.svg)

<span data-ttu-id="ddb85-105">Dans les architectures traditionnelles, le même modèle de données est utilisé pour interroger et mettre à jour une base de données.</span><span class="sxs-lookup"><span data-stu-id="ddb85-105">In traditional architectures, the same data model is used to query and update a database.</span></span> <span data-ttu-id="ddb85-106">Cette approche se révèle simple et efficace pour les opérations CRUD de base.</span><span class="sxs-lookup"><span data-stu-id="ddb85-106">That's simple and works well for basic CRUD operations.</span></span> <span data-ttu-id="ddb85-107">Toutefois, dans le cas des applications plus complexes, elle peut devenir plus difficile à gérer.</span><span class="sxs-lookup"><span data-stu-id="ddb85-107">In more complex applications, however, this approach can become unwieldy.</span></span> <span data-ttu-id="ddb85-108">Par exemple, côté lecture, l’application peut exécuter de nombreuses requêtes, renvoyant des objets de transfert de données de différentes formes.</span><span class="sxs-lookup"><span data-stu-id="ddb85-108">For example, on the read side, the application may perform many different queries, returning data transfer objects (DTOs) with different shapes.</span></span> <span data-ttu-id="ddb85-109">Le mappage d’objets peut alors devenir compliqué.</span><span class="sxs-lookup"><span data-stu-id="ddb85-109">Object mapping can become complicated.</span></span> <span data-ttu-id="ddb85-110">Côté écriture, le modèle peut implémenter une logique métier et de validation complexe.</span><span class="sxs-lookup"><span data-stu-id="ddb85-110">On the write side, the model may implement complex validation and business logic.</span></span> <span data-ttu-id="ddb85-111">Par conséquent, vous risquez d’obtenir un modèle excessivement complexe et surchargé.</span><span class="sxs-lookup"><span data-stu-id="ddb85-111">As a result, you can end up with an overly complex model that does too much.</span></span>

<span data-ttu-id="ddb85-112">Un autre problème potentiel réside dans le fait que les charges de travail de lecture et d’écriture sont souvent asymétriques et présentent des exigences de performances et de mise à l’échelle très différentes.</span><span class="sxs-lookup"><span data-stu-id="ddb85-112">Another potential problem is that read and write workloads are often asymmetrical, with very different performance and scale requirements.</span></span> 

<span data-ttu-id="ddb85-113">CQRS résout ces problèmes en séparant les lectures des écritures dans des modèles distincts et utilise des **commandes** pour mettre à jour les données, et des **requêtes** pour lire les données.</span><span class="sxs-lookup"><span data-stu-id="ddb85-113">CQRS addresses these problems by separating reads and writes into separate models, using **commands** to update data, and **queries** to read data.</span></span>

- <span data-ttu-id="ddb85-114">Les commandes doivent reposer sur les tâches au lieu d’être centrées sur les données</span><span class="sxs-lookup"><span data-stu-id="ddb85-114">Commands should be task based, rather than data centric.</span></span> <span data-ttu-id="ddb85-115">(par exemple, « Réserver une chambre d’hôtel » plutôt que « Définir l’état de réservation sur Réservé »). Les commandes peuvent être placées dans une file d’attente pour faire l’objet d’un traitement asynchrone, au lieu d’être traitées de manière synchrone.</span><span class="sxs-lookup"><span data-stu-id="ddb85-115">("Book hotel room," not "set ReservationStatus to Reserved.") Commands may be placed on a queue for asynchronous processing, rather than being processed synchronously.</span></span>

- <span data-ttu-id="ddb85-116">Les requêtes ne modifient jamais la base de données.</span><span class="sxs-lookup"><span data-stu-id="ddb85-116">Queries never modify the database.</span></span> <span data-ttu-id="ddb85-117">Une requête renvoie un objet de transfert de données qui n’encapsule aucune connaissance du domaine.</span><span class="sxs-lookup"><span data-stu-id="ddb85-117">A query returns a DTO that does not encapsulate any domain knowledge.</span></span>

<span data-ttu-id="ddb85-118">Pour renforcer l’isolation, vous pouvez séparer physiquement les données de lecture des données d’écriture.</span><span class="sxs-lookup"><span data-stu-id="ddb85-118">For greater isolation, you can physically separate the read data from the write data.</span></span> <span data-ttu-id="ddb85-119">Dans ce cas, la base de données de lecture peut utiliser son propre schéma de données optimisé pour les requêtes.</span><span class="sxs-lookup"><span data-stu-id="ddb85-119">In that case, the read database can use its own data schema that is optimized for queries.</span></span> <span data-ttu-id="ddb85-120">Par exemple, elle peut stocker une [vue matérialisée][materialized-view] des données, afin d’éviter des jointures ou des mappages O/RM complexes.</span><span class="sxs-lookup"><span data-stu-id="ddb85-120">For example, it can store a [materialized view][materialized-view] of the data, in order to avoid complex joins or complex O/RM mappings.</span></span> <span data-ttu-id="ddb85-121">Elle peut même utiliser un autre type de magasin de données.</span><span class="sxs-lookup"><span data-stu-id="ddb85-121">It might even use a different type of data store.</span></span> <span data-ttu-id="ddb85-122">Par exemple, il est possible que la base de données d’écriture soit relationnelle, et que la base de données de lecture soit une base de données de documents.</span><span class="sxs-lookup"><span data-stu-id="ddb85-122">For example, the write database might be relational, while the read database is a document database.</span></span>

<span data-ttu-id="ddb85-123">Si vous utilisez des bases de données de lecture et d’écriture distinctes, vous devez les garder synchronisées. En règle générale, vous obtenez ce résultat en faisant en sorte que le modèle d’écriture publie un événement chaque fois qu’il met à jour la base de données.</span><span class="sxs-lookup"><span data-stu-id="ddb85-123">If separate read and write databases are used, they must be kept in sync. Typically this is accomplished by  having the write model publish an event whenever it updates the database.</span></span> <span data-ttu-id="ddb85-124">La mise à jour de la base de données et la publication de l’événement doivent s’effectuer dans une transaction unique.</span><span class="sxs-lookup"><span data-stu-id="ddb85-124">Updating the database and publishing the event must occur in a single transaction.</span></span> 

<span data-ttu-id="ddb85-125">Certaines implémentations de l’architecture CQRS utilisent le [modèle d’approvisionnement en événements][event-sourcing] (ou Event Sourcing).</span><span class="sxs-lookup"><span data-stu-id="ddb85-125">Some implementations of CQRS use the [Event Sourcing pattern][event-sourcing].</span></span> <span data-ttu-id="ddb85-126">Avec ce modèle, l’état de l’application est stocké sous la forme d’une séquence d’événements.</span><span class="sxs-lookup"><span data-stu-id="ddb85-126">With this pattern, application state is stored as a sequence of events.</span></span> <span data-ttu-id="ddb85-127">Chaque événement représente un ensemble de modifications des données.</span><span class="sxs-lookup"><span data-stu-id="ddb85-127">Each event represents a set of changes to the data.</span></span> <span data-ttu-id="ddb85-128">L’état actuel est construit par la relecture des événements.</span><span class="sxs-lookup"><span data-stu-id="ddb85-128">The current state is constructed by replaying the events.</span></span> <span data-ttu-id="ddb85-129">Dans un contexte CQRS, l’un des avantages de l’approvisionnement en événements réside dans le fait que les mêmes événements peuvent être utilisés pour informer les autres composants &mdash; en particulier, pour avertir le modèle de lecture.</span><span class="sxs-lookup"><span data-stu-id="ddb85-129">In a CQRS context, one benefit of Event Sourcing is that the same events can be used to notify other components &mdash; in particular, to notify the read model.</span></span> <span data-ttu-id="ddb85-130">Le modèle de lecture utilise les événements pour créer une capture instantanée de l’état actuel, ce qui se révèle plus efficace pour les requêtes.</span><span class="sxs-lookup"><span data-stu-id="ddb85-130">The read model uses the events to create a snapshot of the current state, which is more efficient for queries.</span></span> <span data-ttu-id="ddb85-131">Toutefois, l’approvisionnement en événements complique la conception.</span><span class="sxs-lookup"><span data-stu-id="ddb85-131">However, Event Sourcing adds complexity to the design.</span></span>

## <a name="when-to-use-this-architecture"></a><span data-ttu-id="ddb85-132">Quand utiliser cette architecture</span><span class="sxs-lookup"><span data-stu-id="ddb85-132">When to use this architecture</span></span>

<span data-ttu-id="ddb85-133">Envisagez d’utiliser CQRS pour les domaines collaboratifs dans lesquels de nombreux utilisateurs accèdent aux mêmes données, en particulier lorsque les charges de travail de lecture et d’écriture sont asymétriques.</span><span class="sxs-lookup"><span data-stu-id="ddb85-133">Consider CQRS for collaborative domains where many users access the same data, especially when the read and write workloads are asymmetrical.</span></span>

<span data-ttu-id="ddb85-134">CQRS n’est pas une architecture de premier niveau qui s’applique à la totalité d’un système.</span><span class="sxs-lookup"><span data-stu-id="ddb85-134">CQRS is not a top-level architecture that applies to an entire system.</span></span> <span data-ttu-id="ddb85-135">Appliquez uniquement CQRS aux sous-systèmes dans lesquels la séparation des lectures et des écritures présente un intérêt manifeste.</span><span class="sxs-lookup"><span data-stu-id="ddb85-135">Apply CQRS only to those subsystems where there is clear value in separating reads and writes.</span></span> <span data-ttu-id="ddb85-136">Dans le cas contraire, vous ajouterez de la complexité sans recevoir aucun avantage en contrepartie.</span><span class="sxs-lookup"><span data-stu-id="ddb85-136">Otherwise, you are creating additional complexity for no benefit.</span></span>

## <a name="benefits"></a><span data-ttu-id="ddb85-137">Avantages</span><span class="sxs-lookup"><span data-stu-id="ddb85-137">Benefits</span></span>

- <span data-ttu-id="ddb85-138">**Mise à l’échelle de manière indépendante**.</span><span class="sxs-lookup"><span data-stu-id="ddb85-138">**Independently scaling**.</span></span> <span data-ttu-id="ddb85-139">CQRS permet de mettre à l’échelle les charges de travail de lecture et d’écriture indépendamment, et peut contribuer à réduire les contentions de verrouillage.</span><span class="sxs-lookup"><span data-stu-id="ddb85-139">CQRS allows the read and write workloads to scale independently, and may result in fewer lock contentions.</span></span>
- <span data-ttu-id="ddb85-140">**Schémas de données optimisés**.</span><span class="sxs-lookup"><span data-stu-id="ddb85-140">**Optimized data schemas.**</span></span>  <span data-ttu-id="ddb85-141">Le côté lecture peut utiliser un schéma optimisé pour les requêtes, tandis que le côté écriture utilise un schéma optimisé pour les mises à jour.</span><span class="sxs-lookup"><span data-stu-id="ddb85-141">The read side can use a schema that is optimized for queries, while the write side uses a schema that is optimized for updates.</span></span>  
- <span data-ttu-id="ddb85-142">**Sécurité**.</span><span class="sxs-lookup"><span data-stu-id="ddb85-142">**Security**.</span></span> <span data-ttu-id="ddb85-143">Il est plus facile de s’assurer que seules les entités de domaine adéquates effectuent des écritures sur les données.</span><span class="sxs-lookup"><span data-stu-id="ddb85-143">It's easier to ensure that only the right domain entities are performing writes on the data.</span></span>
- <span data-ttu-id="ddb85-144">**Séparation des problèmes**.</span><span class="sxs-lookup"><span data-stu-id="ddb85-144">**Separation of concerns**.</span></span> <span data-ttu-id="ddb85-145">Le fait de séparer le côté lecture du côté écriture peut produire des modèles plus faciles à gérer et plus flexibles.</span><span class="sxs-lookup"><span data-stu-id="ddb85-145">Segregating the read and write sides can result in models that are more maintainable and flexible.</span></span> <span data-ttu-id="ddb85-146">La plus grande partie de la logique métier complexe est placée dans le modèle d’écriture.</span><span class="sxs-lookup"><span data-stu-id="ddb85-146">Most of the complex business logic goes into the write model.</span></span> <span data-ttu-id="ddb85-147">Le modèle de lecture peut être relativement simple.</span><span class="sxs-lookup"><span data-stu-id="ddb85-147">The read model can be relatively simple.</span></span>
- <span data-ttu-id="ddb85-148">**Requêtes simplifiées**.</span><span class="sxs-lookup"><span data-stu-id="ddb85-148">**Simpler queries**.</span></span> <span data-ttu-id="ddb85-149">En stockant une vue matérialisée dans la base de données de lecture, l’application peut éviter les jointures complexes lors de l’interrogation.</span><span class="sxs-lookup"><span data-stu-id="ddb85-149">By storing a materialized view in the read database, the application can avoid complex joins when querying.</span></span>
